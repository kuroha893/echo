# Echo 参赛介绍文档

## 1. 项目概述

Echo 是一个面向实时陪伴场景的低延迟 Agent Runtime。它不是单一的大模型聊天壳，也不是只能做问答的普通桌宠，而是一个围绕“实时陪伴”构建的完整运行时系统：既能进行自然对话，也能通过 Live2D 形象、语音合成、桌面字幕、环境感知、多角色故事模式等能力，形成更具存在感、表达力和沉浸感的人机交互体验。

项目整体遵循 runtime-first 的设计思路，核心目标不是堆砌单点功能，而是围绕真实交互过程中的几个关键问题展开：

- 如何让 AI 在桌面上“活起来”，而不是只出现在一个输入框里。
- 如何让交互足够低延迟，在用户输入后快速给出可感知反馈。
- 如何让角色具有表情、动作、语气与状态变化，而不是单纯输出文本。
- 如何让系统具备可扩展性，为后续记忆、插件、感知、多角色协作等能力留出稳定接口。

Echo 当前已经形成一条完整可演示闭环：桌面角色形象、聊天窗、字幕窗、Web 控制台、模型库管理、云端主模型、可选本地快模型、语音克隆、环境感知、多角色故事模式都已经能够协同工作。

## 2. 项目定位与应用价值

Echo 关注的是“实时陪伴型智能体”的实际落地形态。在传统聊天机器人中，交互通常停留在“文字输入—文字输出”这一层，缺少角色表达、持续陪伴和桌面融入感。Echo 试图解决以下问题：

- 让 AI 从“网页里的对话框”变成“桌面上的角色化存在”。
- 让文本、语音、表情、动作、字幕、场景状态形成统一协同。
- 让角色不仅被动回答，还可以基于环境变化做有限度、可控的主动评论。
- 让多角色共同参与叙事，探索更丰富的故事式交互体验。

应用场景包括但不限于：

- 桌面陪伴与情绪互动。
- 剧情式陪伴与角色扮演。
- 学习、创作、写作时的智能陪伴。
- 面向未来的插件扩展场景，例如网页理解、工具协同、长期记忆等。

## 3. 基本功能介绍

### 3.1 桌面三窗口陪伴交互

Echo 采用角色窗、聊天窗、字幕窗三窗口协同的桌面形态。

- 角色窗负责展示 Live2D 角色形象、动作、表情与说话状态。
- 聊天窗负责显示会话历史、输入区、图片输入与会话切换。
- 字幕窗负责在桌面上高可见度展示当前说话内容，提升陪伴感与可读性。

这三者并不是孤立界面，而是由统一运行时驱动：当模型生成回复时，文本会同步进入聊天窗，语音会进入播放链路，表情与动作指令会进入渲染器，字幕则以更适合桌面观看的方式显示在气泡层。



### 3.2 Live2D 角色驱动与表情动作联动

Echo 内置 Live2D 角色库，用户可以自由往角色库中加入自己的live2d模型并导入echo，不依赖绝对路径，保证模型切换与持久化更稳定。

角色不仅能静态显示，还支持：

- 状态切换，如 idle、listening、thinking、speaking。
- 表情切换，如 smile、thinking 等表达。
- 动作播放，如点头、惊讶、摇头等动作。
- 基于语音播放驱动的嘴型同步。
- 可选的鼠标视线追踪，让角色随鼠标产生注视感。

系统会对模型资源进行严格校验，确保 Cubism 资源包中的 model3.json、motion3.json、exp3.json、纹理等文件都位于受控资源目录中，避免资源越界与非法引用。并且每个live2d都可以配置独立的人格文档，每个人格的聊天记录也都相互隔离

### 3.3 聊天会话管理

Echo 的聊天窗口并不只是一个简单输入框，而是带有会话管理能力的桌面对话面板。

- 会话列表支持查看当前会话。
- 当前活动会话会被明确高亮。
- 支持创建、切换、删除、分叉会话。
- 支持查看完整 transcript 历史。
- 支持流式显示模型输出过程。
- 支持图片输入，为后续多模态理解提供入口。

分叉会话是一个很实用的小功能：用户可以从已有对话中截取某一段继续延展，形成新的会话分支，用于创作、角色对话实验或多版本探索。

### 3.4 实时字幕与桌面气泡

Echo 的字幕窗并非传统意义上的聊天弹窗，而是专门为桌面场景设计的高可视浮层。

- 字幕窗默认底部居中显示，适合桌面观感。
- 字幕支持大字号、描边、阴影和半透明背景，提升复杂背景下的可读性。
- 字幕支持最小化、关闭和拖拽移动。
- 在故事模式下，字幕还能按角色区分颜色，辅助用户理解当前说话者。

这一设计兼顾了美观与实用：角色在说话时，用户无需始终盯着聊天窗，也能从桌面中央区域快速获取信息。

### 3.5 语音合成与语音克隆

Echo 集成了 Qwen TTS 语音链路，并进一步支持语音克隆。用户可以在 Web 控制台中直接拖入本地音频样本或从磁盘选择音频文件，完成语音注册与生成。

当前设计具有以下特点：

- 配置界面将主模型设置、TTS 设置、语音注册放在同一控制平面中，减少跳转成本。
- 语音注册支持拖拽上传，交互更直观。
- 上传前会做音频类型校验，支持 wav、mp3、m4a、ogg、flac、aac、webm 等常见格式。
- 注册完成后会生成 voice profile，并进入本地受控音色库，后续可直接复用。

这使得 Echo 不只是“会说话”，而是可以进一步贴近角色设定和用户偏好，增强人物感与沉浸感。

### 3.6 环境感知与主动评论

Echo 具备桌面环境感知能力，可以在用户使用电脑的过程中，对前台应用、窗口标题、部分视觉变化做轻量分析，并在适当时机给出主动评论。

这项功能并不是无节制地“打扰用户”，而是带有多层限制策略：

- 只在系统空闲、角色不在说话/思考时触发。
- 对同一场景、同一应用、短时间内重复内容设置冷却时间。
- 对重复评论进行去重，避免“反复说同一句”。
- 支持从托盘中一键开启或关闭。
- 在工作模式下优先使用结构化上下文，避免过度依赖截图。

这让角色从被动聊天对象，向“具有有限主动性的桌面陪伴体”转变，是项目陪伴感的重要来源之一。

### 3.7 多角色故事模式

多角色故事模式是 Echo 的标志性功能之一。系统可以同时管理多个角色，由旁白与角色共同推进故事进程，形成持续演化的互动叙事。

当前故事模式支持：

- 创建 story thread。
- 配置多个 cast member。
- 为不同角色绑定模型、persona、音色。
- 维护统一时间线、场景状态与旁白状态。
- 保存和读取故事存档。
- 在需要时为用户提供候选行动或剧情选项。

与普通“多机器人群聊”不同，Echo 的故事模式更像一个可控的叙事运行时：系统区分 narrator 与 cast，维护 stage state、timeline、narrator state 等结构化状态，并通过桌面字幕颜色、时间线展示、场景信息等方式，让用户理解故事当前处于什么阶段。

## 4. 技术架构与技术栈

### 4.1 整体架构

Echo 采用 Electron + Python Host 的双端协同架构。

- Electron 主进程负责桌面窗口管理、系统托盘、控制平面、环境感知、与渲染层交互。
- Python Host 负责会话管理、运行时服务编排、协议校验、LLM/TTS 服务组装。
- Renderer 层负责角色渲染、聊天界面、字幕气泡与控制台页面。
- Web Control Plane 负责配置主模型、TTS、语音注册、模型与运行状态查看。

Electron 与 Python Host 之间采用 JSON over stdio 的桥接方式，避免了过重的本地服务依赖，也便于保持边界清晰。

### 4.2 前端与桌面技术栈

- Electron 36.4.0：负责桌面应用外壳、窗口管理与系统能力接入。
- Node.js ESM：主进程与桌面桥接层均采用现代 ESM 模块。
- PixiJS 6.5.10：负责高性能 2D 渲染。
- pixi-live2d-display 0.4.0：实现 Live2D Cubism 模型加载与驱动。
- HTML/CSS/JavaScript：构建聊天窗、字幕窗、控制台与 Web Control Plane。

在 UI 设计上，Echo 并没有采用大而重的前端框架，而是保留轻量、可控的原生渲染结构，以降低桌面环境下的额外开销。

### 4.3 后端与运行时技术栈

- Python 3 环境：承载 companion host 与运行时主逻辑。
- Pydantic v2：用于协议模型、事件载荷和会话持久化的强类型校验。
- packages/protocol：定义事件、状态机、反馈规则等 typed contracts。
- packages/runtime：负责 session 生命周期、catalog 持久化、桌面 companion service。
- packages/orchestrator：负责调度、打断、安全切换、快速反应与主回答编排。
- packages/llm：封装云端与本地 LLM provider。
- packages/tts：封装语音合成、音色配置、语音克隆。
- packages/renderer：封装渲染器命令与桌面桥接协议。

整个系统非常强调“协议先行”和“强类型约束”，减少复杂系统中常见的状态漂移和隐式耦合问题。

### 4.4 模型与语音能力栈

- 云端主模型：使用 OpenAI Responses API 作为 production primary 路径。
- 本地快模型：支持 OpenAI-compatible 本地推理端点，可作为可选加速层。
- 语音合成：集成 Qwen TTS。
- 语音克隆：支持从参考音频生成 voice profile 并写入本地音色库。

这种“云端主模型 + 本地快模型”的双轨结构，使 Echo 兼顾了回答质量与交互速度。

### 4.5 渲染与交互细节

Echo 的桌面窗口不是普通白底应用，而是专门为陪伴场景打磨的轻浮层结构：

- 角色窗默认尺寸为 460×620，透明、无边框、常驻最前、可调整大小。
- 聊天窗默认尺寸为 460×620，无边框、毛玻璃背景、常驻最前。
- 字幕窗默认尺寸为 1440×260，位于屏幕底部居中区域，用于大字号字幕展示。
- 聊天窗与角色窗之间默认保持 12px 间距，自动根据显示区域计算布局。
- 字幕窗会根据显示器工作区自适应宽度与高度，保证在不同屏幕上都能保持良好可视性。

这些细节体现了项目在“真实桌面使用体验”层面的打磨，而不只是做功能展示。

## 5. 创新之处与启发来源

### 5.1 创新点一：从“聊天机器人”走向“桌面陪伴运行时”

Echo 最大的创新不是单一功能，而是产品形态的升级。我们没有把项目做成单纯的聊天框，而是把角色显示、文字交互、桌面字幕、语音播放、状态驱动、系统托盘与控制平面统一到一个实时运行时中。

这意味着 Echo 的目标不是“回答一个问题”，而是让角色始终作为一个可感知、可表达、可切换状态的桌面存在体参与用户的使用过程。

启发来源：

- 受 open-yachiyo 在桌面 Live2D 呈现与 Pixi 集成上的成熟经验启发。
- 受 AIRI 在虚拟角色陪伴感、角色化交互思路上的启发。
- 但 Echo 在此基础上进一步强调运行时协议、状态机、桌面多窗口协同与低延迟编排，这部分是项目自身的核心设计。

### 5.2 创新点二：低延迟双轨模型路由

Echo 将大模型调用拆分为不同 route kind，例如 intent_routing、quick_reaction、primary_response 等，让本地快模型与云端主模型各司其职：

- 本地快模型负责低延迟判断和轻量响应。
- 云端主模型负责高质量完整回答。
- 系统可根据任务类型决定走哪条路径。

这种设计有效平衡了响应速度与回答质量，解决了桌面陪伴系统中“太慢就不像陪伴、太快又不够聪明”的常见矛盾。

启发来源：

- 来源于我们对实时陪伴场景的需求分析，以及对传统单路大模型交互延迟问题的反思。
- 同时参考了本地推理与云端推理协同的工程实践，但 Echo 通过统一 route 与 provider profile 机制将其系统化。

### 5.3 创新点三：文本、语音、表情、动作四路联动

Echo 不把模型输出当作单纯文本，而是把它拆分为多条可协同的表现层信号：

- 文本进入聊天窗。
- 字幕进入桌面气泡层。
- 语音进入 TTS 与播放链路。
- 表情与动作指令进入 Live2D 渲染器。

这使角色的“说话”不只是字面意义上的发声，而是形成完整的角色表达。对于桌面陪伴系统来说，这种多通道联动比单纯 TTS 或单纯头像切换更自然，也更能塑造角色感。

启发来源：

- 受虚拟主播与数字角色系统中“表现层同步”的思路启发。
- 也参考了 open-yachiyo、AIRI 等项目在虚拟角色表达上的长处。
- Echo 在工程上通过解析器、渲染命令与音频播放链路进行了统一落地。

### 5.4 创新点四：多角色故事模式

多角色故事模式突破了“一问一答式”陪伴形态，让 AI 角色之间可以共享同一故事线程，在旁白引导下共同推进情节。

相较于普通多智能体聊天，Echo 的故事模式更强调：

- 旁白与角色职责分离。
- 时间线与场景状态可视化。
- 角色发言与桌面表现同步。
- 用户可在关键节点介入，影响剧情方向。

这类能力对互动叙事、角色陪伴、剧情演示等场景具有较强创新价值。

启发来源：

- 受 AgentGal 等多角色叙事设计的启发。
- 结合 Echo 现有桌面陪伴架构进行改造，不是把多代理逻辑简单叠加，而是将其嵌入统一 thread、timeline 与 desktop presentation 中。

### 5.5 创新点五：有限主动性的环境感知评论

很多 AI 助手只能等待输入，而 Echo 已经迈出一步，尝试通过桌面环境感知实现有限主动评论。它不会无规则打断，而是在冷却、空闲、去重、上下文判定之后，适度地对用户当前桌面状态做出回应。

这使角色更像陪伴者，而不只是工具接口。

启发来源：

- 受桌宠、虚拟陪伴体、环境感知智能体等方向启发。
- Echo 特别强调可控性与工程约束，因此加入了节流、去重、隐私边界与托盘开关机制。

## 6. UI 与交互打磨

### 6.1 桌面角色窗

角色窗采用透明无边框设计，使角色仿佛直接悬浮在桌面上，而不是被装进一个厚重的应用窗口。为了兼顾可玩性与可控性，角色窗还支持：

- Resize Mode：进入后会显示高亮描边边框。
- Layout Tuner：可微调 X/Y 偏移与缩放比例。
- Click-through：可切换鼠标穿透，避免挡住用户操作。
- Drag Zone 可视化：帮助用户理解拖拽区域。

这些能力很小，但非常影响日常使用体验，体现了桌宠产品的“可长期共存”思路。

### 6.2 字幕窗视觉设计

字幕窗选择了高识别度的 M PLUS Rounded 1c 字体，配合大字号、描边、阴影、半透明胶囊面板与平滑过渡，兼顾角色感和信息可读性。

具体设计细节包括：

- 默认 46px 大号字幕，移动端/小尺寸下降到 32px。
- 180ms 的透明度过渡，观感柔和。
- 通过描边和投影增强复杂背景下的可读性。
- 支持字幕条拖拽、最小化与关闭。

对于桌面使用来说，这些细节让字幕真正成为“可读、可看、可长期存在”的信息层。

### 6.3 聊天窗交互设计

聊天窗采用半透明深色面板、圆角卡片、毛玻璃背景与清晰的消息分层：

- 用户消息与助手消息使用不同底色。
- 流式消息带有渐显效果，增强“正在生成”的动态反馈。
- 顶部有 runtime 状态与 transcript 计数，让系统状态更透明。
- 底部输入框支持多行输入，回车发送、Shift+Enter 换行。
- 内置 Debug Panel，用于显示本地控制平面的调试流。

这套设计既有桌面软件的稳定感，也保留了 AI 对话产品所需的实时反馈感。

## 7. 方便实用的小巧思功能

除了大功能之外，Echo 还有许多非常实用、适合展示产品打磨程度的小功能：

- 托盘双击可一键显示全部窗口。
- 托盘菜单可直接打开角色窗、聊天窗、控制台。
- 托盘中可直接切换环境感知与鼠标视线追踪。
- 聊天窗支持图片输入，为多模态交互保留了自然入口。
- 角色窗支持鼠标穿透，在日常桌面使用中不妨碍其他软件操作。
- 字幕窗支持拖拽与最小化，减少占屏干扰。
- 模型选择持久化以 model_key 为核心，而非脆弱的本地绝对路径。
- 语音注册支持直接拖拽音频，而非手动填写文件路径。
- 故事模式为不同角色预设字幕色与时间线色，降低多角色场景下的认知成本。

这些功能单独看都不算“震撼”，但它们共同决定了一个系统能否从 Demo 走向可持续使用的产品形态。

## 8. 工程特色与可靠性设计

Echo 在工程上非常重视协议清晰、边界明确与强校验，主要体现在：

- 采用 Pydantic v2 对协议与持久化结构做严格验证。
- 采用显式状态机定义 idle、listening、thinking、speaking、interrupted、error 等状态。
- Electron 与 Python Host 边界清晰，避免桌面层和运行时逻辑混杂。
- 所有核心能力按包边界拆分，便于后续扩展记忆、插件、STT、工具链等能力。
- 坚持 fail fast 原则，尽量避免隐性兜底导致的问题被掩盖。

对于比赛评审而言，这意味着 Echo 不是“只跑得起来的原型”，而是一套具有清晰工程方法论的系统。

## 9. 项目亮点总结

综合来看，Echo 的亮点可以概括为以下几点：

- 形态创新：从聊天机器人升级为桌面陪伴运行时。
- 系统创新：将文本、语音、Live2D、字幕、状态机统一到同一条实时链路中。
- 交互创新：三窗口桌面协同 + Web Control Plane + 多角色故事模式。
- 工程创新：本地快模型与云端主模型双轨路由，兼顾速度与质量。
- 产品打磨：字幕窗、托盘、穿透、拖拽、布局微调、会话分叉等细节完整。

Echo 所展示的不只是某一项模型能力，而是“如何把大模型真正做成一套有陪伴感、有角色感、有产品完成度的桌面系统”。

## 10. 比赛展示建议

如果需要进行路演或答辩展示，建议按照以下顺序组织演示：

1. 先展示三窗口桌面形态，说明 Echo 不是传统聊天框，而是完整桌面陪伴体。
2. 展示聊天输入后，角色、字幕、语音、聊天记录同步变化的全过程。
3. 展示 Config v2 页面，说明主模型、语音克隆、本地快模型、模型库管理是如何统一配置的。
4. 展示环境感知的托盘开关与一次轻量主动评论。
5. 展示多角色故事模式，突出时间线、角色分工和剧情推进。
6. 最后用系统架构图总结 Echo 的 runtime-first 设计与工程可扩展性。

## 11. 可直接使用的结尾总结

Echo 试图回答一个问题：在大模型时代，AI 是否只能停留在聊天框里？

我们的答案是，不是。AI 可以成为桌面上的角色、会说话的陪伴者、带有表情动作的数字生命体，也可以成为一个具备扩展潜力的实时运行时系统。Echo 正是在这个方向上的一次系统化探索。

它把 Live2D、桌面多窗口、语音克隆、低延迟模型路由、环境感知与多角色叙事整合到同一平台中，既体现了工程能力，也体现了交互设计与产品思考。对于计算机设计大赛而言，Echo 的价值不仅在于“功能多”，更在于它展示了一条从模型能力走向完整交互产品的清晰路径。
